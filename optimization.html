ProcessAnalys - Optimering

Inledning

Jag börjar med det uppenbara att banta index.html och styles.css filerna på det som inte behövs. Bortkommenterad kod tar jag bort, fonter som inte används, upprepad kod som gör samma sak etc. Citat från mdn-web-docs:

Remove unnecessary styles: This may sound obvious, but it is surprising how many developers forget to clean up unused CSS rules that were added to their stylesheets during development and ended up not being used.
- Mdn-Web-Docs (CSS Performance Optimization - Learn Web Development | MDN, 2023)

Jag ändrar sedan så att alla fonter som används är inom samma link href, tidigare hade jag flera länkar för varje enskild font. Inom script-tagen för font-awesome lägger jag till async-attributet som låter html bli tolkat medans scriptet hämtas vilket kan förbättra sidans laddningshastighet. Jag tar bort bilder som inte används. Jag tar bort css-filer som egentligen inte behövs i mitt fall då de inte gör någon faktiskt skillnad på utseendet när jag provar att kommentera bort koden. Jag hade en css-reset fil från Eric Meyer (CSS Tools: Reset CSS, 2019) Som inte behövs. Jag lägger till media-queries i botten av style.css filen istället för att ha en separat fil för detta. Jag tar även bort fonts.css den innehåller bara en font-face regel som jag istället kan lägga i toppen av style.css. Jag kollar efter upprepning i css.style filen och ser till exempel att jag skrivit nav a {color: #fff) och sedan senare #header a {color: #fff) dessa kan jag kombinera till nav a, #header a {color: #fff) istället. Detta gör jag på fler ställen så att upprepning minskas och därmed blir filen mindre. Jag tar bort whitespaces mellan kommentarer och delar av css-kod.

BILDOPTIMERING

Det finns flera sätt att optimera bilder som är en vanlig bov bakom sega laddningtider för en webbsida pga dess storlek som ska hämtas ner från klient till server. Jag börjar lätt med att lägga till lazy-loading som görs genom att lägga till ett loading-attribut på ett img-element som i sin tur sätts till ”lazy”. Här kommer ett citat från mdn-web-docs som förklarar varför jag beslutat mig för detta.

Very often, webpages contain many images that contribute to data-usage and how fast a page can load. Most of those images are off-screen (non-critical), requiring a user interaction, like scrolling, in order to view them.
- Mdn-Web-Docs (Lazy Loading - Web Performance | MDN, n.d.) Det är alltså onödigt att bilder som ligger utanför det användaren ser på skärmen ska laddas in direkt och just detta motverkar jag med lazy-loading. Nästa steg jag tar för att optimera min webbsida är att försöka få ner storleken på mina bilder. Detta kan jag göra genom kompression. Förtydligande:
Bildkompression omfattar metoder för att komprimera i huvudsak två-dimensionella digitala bilder i syfte att minska datavolymen för lagring och/eller överföring.
- Wikipedia (Bildkompression, 2023) För att göra detta finns en rad olika program/konverterare både desktop och webb-baserade. Jag följer Christians exempel (Lärare inom kursen ”Avancerad Webbutveckling med HTML och CSS” (AWH201)) och använder mig av squoosh.app. (Squoosh, n.d.) I och med detta lyckas jag få ner storleken på bildfilerna en hel del genom att offra en viss bildkvalitet som inte är speciellt noterbar, det kan det vara värt för att få webbsidan att ladda snabbare.

VILKEN TEKNIK?

Jag börjar fundera över vilken av teknikerna BEM, SMACSS eller OOCSS som jag ska använda mig av. Jag googlar runt för att se vad som används mest, vad som kan tänka sig vara populärast alternativt bäst. Jag får fram svar och indikerar på att något skulle vara bättre än det andra utan att det är i vilket sammanhang man ska använda någon av dessa tekniker som avgör vad som är passande. Jag kom över en blog-plost: Organize-css-modular-architecture vilket förde mig emot att använda BEM. (Organize CSS with a Modular Architecture: OOCSS, BEM, SMACSS, 2022). Jag läste sedan en blogpost/artikel av Harry Roberts och kände mig redo att sätta mig in I tekniken efter detta citat:

I’d strongly encourage others to consider adopting it as well because it just makes everything a lot tighter and more connected, making code easier to maintain by teams, or even just by yourself a few months down the line.
– Harry Roberts (MindBEMding – Getting Your Head ’Round BEM Syntax – CSS Wizardry–Web Performance Optimisation, 2013) Jag blev än mer styrkt I att använda BEM genom detta citat från deras hemsida getbem.com
The reason I choose BEM over other methodologies comes down to this: it is less confusing than the other methods (i.e. SMACSS) but still provides us the good architecture we want (i.e. OOCSS) and with a recognizable terminology.
- Mark McDonnell (Maintainable CSS with BEM (BEM — Block Element Modifier))

BEM

BEM står för Block, Element, Modifer och är en metodik och namnkonvention som används för att kunna implementera hanterbar CSS som är skalbar och fungerar väl för underhåll och förändring under ett projekt. BEM stylar HTML-element och komponenter på ett modulärt och strukturerat sätt.

Varför ska man använda BEM?

Om man vill implementera en ny stil för en komponent så ser man tydligt vilka modifierare och barn (modifiers/children) som redan finns. Man kan genom BEM snabbt få en uppfattning om vilket element som är beroende av ett annat genom namnkonventionen. Som exempel att .btn__highlight hör till/är beroende av .btn. Designers och utvecklare kan konsekvent namnge komponenter för enklare kommunikation mellan teammedlemmar. BEM ger alla i ett projekt ett syntax(regler) att följa som gör att de håller sig på samma bana och undviker konflikt mellan olika sätt att förhålla sig till CSS inom ett projekt. Det effektiverar även projektprocessen genom att alla håller sig från början till samma regler. Harry Roberts förklarar det stora problemet med att inte använda sig av en metodik som BEM, man saknar helt enkelt självförtroende eftersom man inte vet vad saker gör, det blir övermäktigt och man vågar inte ändra på något.

This is the main reason we end up with bloated code bases, full of legacy and unknown CSS that we daren’t touch. We lack the confidence to be able to work with and modify existing styles because we fear the consequences of CSS’ globally operating and leaky nature. Almost all problems with CSS at scale boil down to confidence (or lack thereof): People don’t know what things do any more. People daren’t make changes because they don’t know how far reaching the effects will be.
- Harry Roberts (BEM 101, 2015)

Lighthouse

Lighthouse är ett automatiserat verktyg med öppen källkod för att förbättra kvaliteten på webbsidor. Verktyget kan köras mot vilken webbsida som helst både offentliga såväl som de som kräver autentisering. Lighthouse kör granskningar av prestanda, tillgänglighet, progressiva webbappar, SEO och mer därtill. Verktyget kan köras genom DevTools i webbläsaren Chrome, från kommandoraden eller som nodmodul.

TESTRESULTAT MED LIGHTHOUSE: ICKE-OPTIMERAD WEBBSIDA

Resultatet hos den icke-optimerade webbsidan lämnar utrymme till förbättring med ett resultat på 84 i performance. Enbart kompressionen av bilderna och tömmandet av onödigt innehåll i koden kommer förhoppningsvis att få upp denna siffra. Ett screenshot på network taben i dev-tools som också kan vara relevant att granska vid optimering av en webbsida.
